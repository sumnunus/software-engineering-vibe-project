# 요구사항 분석

## 문서 개요

- 프로젝트명: 외출 준비 통합 도우미 앱
- 작성일: 2026-05-24
- 작성 목적: 바이브코딩 프로젝트의 요구사항을 소프트웨어공학 프로세스에 따라 분석하고, 이후 설계, 구현, 테스트 단계의 기준으로 사용한다.
- 작성 방식: 개인 프로젝트 수행자가 문제 상황을 정의하고, AI와의 토의를 통해 요구사항 누락 여부와 우선순위를 점검한다.

## 문제 정의

아침 외출 준비 시간에는 준비물 확인, 날씨 확인, 옷차림 판단, 교통 정보 확인, 출발 시각 관리가 동시에 필요하다. 그러나 현재 사용자는 여러 앱을 오가며 정보를 확인해야 하므로 시간이 낭비되고, 준비물을 빠뜨리거나 출발 시간을 놓칠 가능성이 있다.

이 프로젝트는 외출 전 필요한 정보를 하나의 화면에서 확인할 수 있는 통합 대시보드를 제공하여, 사용자가 바쁜 상황에서도 빠르게 준비 상태를 점검하도록 돕는 것을 목표로 한다.

## 대상 사용자

주요 사용자는 등교, 출근, 약속 등 정해진 시간에 외출해야 하는 개인 사용자이다. 특히 아침 시간처럼 준비 시간이 짧고 동시에 확인해야 할 정보가 많은 상황을 주요 사용 맥락으로 설정한다.

## 사용자 페르소나

### Persona 1. 아침 등교를 준비하는 대학생

- 상황: 수업 시작 시간에 맞춰 집을 나서야 한다.
- 문제: 날씨 앱, 지도/교통 앱, 메모 앱을 각각 확인해야 해서 번거롭다.
- 목표: 나가기 전에 챙길 물건과 교통 정보를 한 번에 확인하고 싶다.

### Persona 2. 출근 전 시간이 부족한 직장인

- 상황: 출근 준비를 하면서 버스 도착 시간과 날씨를 빠르게 확인해야 한다.
- 문제: 옷차림을 결정하고 준비물을 확인하는 과정에서 출발 시간이 늦어질 수 있다.
- 목표: 현재 준비 상태와 남은 시간을 한 화면에서 보고 싶다.

## 핵심 사용자 시나리오

1. 사용자는 앱을 실행한다.
2. 앱은 오늘 날짜와 외출 준비 대시보드를 보여준다.
3. 사용자는 외출 전 챙겨야 할 준비물 목록을 확인하고 완료 여부를 체크한다.
4. 사용자는 현재 날씨와 기온을 확인한다.
5. 앱은 기온에 맞는 간단한 옷차림 추천을 제공한다.
6. 사용자는 사전에 등록한 버스의 도착까지 남은 시간을 확인한다.
7. 사용자는 설정한 외출 시각까지 남은 시간을 확인한다.
8. 사용자는 모든 정보를 바탕으로 외출 준비 상태를 판단한다.

## 범위

### 포함 범위

- 단일 대시보드 화면
- 준비물 투두 리스트
- 날씨 및 기온 표시
- 기온 기반 옷차림 추천
- 버스 도착 정보 표시
- 외출 시각 및 남은 시간 표시
- 기본 테스트 및 화면 캡처

### 제외 범위

- 사용자 계정 및 로그인 기능
- 실제 결제 또는 상용 서비스 연동
- 복잡한 알림 스케줄링
- 여러 사용자 간 데이터 공유
- 장기적인 날씨 예보 분석

## 요구사항 우선순위 기준

- Must: 프로젝트 목표 달성을 위해 반드시 필요한 기능
- Should: 핵심 경험을 보강하지만 일정에 따라 조정 가능한 기능
- Could: 시간이 남을 경우 구현할 수 있는 개선 기능
- Won't: 이번 프로젝트 범위에서는 제외하는 기능

## 기능 요구사항

| ID | 요구사항 | 우선순위 | 설명 | 수용 기준 |
| --- | --- | --- | --- | --- |
| FR-01 | 대시보드 화면 제공 | Must | 사용자는 앱 실행 시 외출 준비 정보를 한 화면에서 확인할 수 있어야 한다. | 투두, 날씨, 옷차림, 버스, 외출 시간 영역이 한 화면에 표시된다. |
| FR-02 | 준비물 투두 목록 표시 | Must | 사용자는 나가기 전 챙겨야 할 준비물을 목록으로 볼 수 있어야 한다. | 기본 준비물 항목이 표시되고 완료 여부를 확인할 수 있다. |
| FR-03 | 준비물 완료 체크 | Must | 사용자는 각 준비물 항목을 완료 처리할 수 있어야 한다. | 체크한 항목은 완료 상태로 시각적으로 구분된다. |
| FR-04 | 오늘 날씨 및 기온 표시 | Must | 사용자는 현재 날씨와 기온을 확인할 수 있어야 한다. | 대시보드에 날씨 상태와 기온이 표시된다. |
| FR-05 | 기온별 옷차림 추천 | Must | 앱은 기온에 따라 적절한 옷차림을 추천해야 한다. | 기온 구간에 맞는 추천 문구가 표시된다. |
| FR-06 | 외출 시각 표시 | Must | 사용자는 설정된 외출 시각을 확인할 수 있어야 한다. | 대시보드에 외출 예정 시각이 표시된다. |
| FR-07 | 외출까지 남은 시간 표시 | Must | 앱은 현재 시각 기준으로 외출까지 남은 시간을 보여줘야 한다. | 남은 시간이 분 단위로 표시되고 시간이 지나면 갱신된다. |
| FR-08 | 버스 도착 정보 표시 | Should | 사용자는 사전에 등록한 버스의 도착까지 남은 시간을 확인할 수 있어야 한다. | 등록된 버스 번호 또는 노선명과 도착 예정 시간이 표시된다. |
| FR-09 | mock 데이터 기반 버스 정보 | Should | 실제 API 연동이 어려운 경우 mock 데이터로 버스 도착 정보를 표현할 수 있어야 한다. | API 없이도 버스 정보 영역이 정상적으로 표시된다. |
| FR-10 | 기본 설정값 제공 | Should | 초기 실행 시에도 사용자가 기능을 확인할 수 있도록 예시 데이터가 제공되어야 한다. | 기본 준비물, 외출 시각, 버스 정보가 예시로 표시된다. |
| FR-11 | 준비물 항목 추가 | Could | 사용자는 준비물 항목을 직접 추가할 수 있다. | 입력한 항목이 목록에 추가된다. |
| FR-12 | 준비물 항목 삭제 | Could | 사용자는 불필요한 준비물 항목을 삭제할 수 있다. | 삭제한 항목이 목록에서 제거된다. |
| FR-13 | 외출 시각 수정 | Could | 사용자는 외출 예정 시각을 변경할 수 있다. | 변경된 시각을 기준으로 남은 시간이 다시 계산된다. |

## 비기능 요구사항

| ID | 요구사항 | 우선순위 | 설명 | 수용 기준 |
| --- | --- | --- | --- | --- |
| NFR-01 | 사용성 | Must | 아침의 바쁜 상황에서도 빠르게 읽을 수 있어야 한다. | 주요 정보가 한 화면에서 명확히 구분되어 보인다. |
| NFR-02 | 응답성 | Must | 사용자 조작에 즉시 반응해야 한다. | 체크박스 클릭, 시간 갱신 등 기본 조작이 지연 없이 동작한다. |
| NFR-03 | 가독성 | Must | 날씨, 시간, 준비물 정보가 작은 화면에서도 읽기 쉬워야 한다. | 모바일 또는 좁은 화면에서도 텍스트가 겹치지 않는다. |
| NFR-04 | 유지보수성 | Should | 기능별 코드가 지나치게 섞이지 않아야 한다. | 투두, 날씨, 버스, 시간 표시 로직을 구분하여 관리한다. |
| NFR-05 | 테스트 가능성 | Should | 주요 기능은 테스트 항목으로 검증할 수 있어야 한다. | 기능별 테스트 케이스가 `docs/test_report.md`에 정리된다. |
| NFR-06 | 확장성 | Could | 추후 실제 API 연동으로 확장할 수 있어야 한다. | mock 데이터와 실제 API 데이터를 교체하기 쉬운 구조를 고려한다. |

## 데이터 요구사항

| 데이터 | 필드 예시 | 설명 |
| --- | --- | --- |
| 준비물 항목 | id, title, completed | 외출 전 챙겨야 할 항목 |
| 날씨 정보 | temperature, condition | 현재 기온과 날씨 상태 |
| 옷차림 추천 | temperatureRange, recommendation | 기온 구간별 추천 문구 |
| 버스 정보 | routeName, stopName, minutesLeft | 등록된 버스와 도착 예정 시간 |
| 외출 시각 | targetTime | 사용자가 설정한 외출 예정 시각 |

## 제약사항 및 가정

- 프로젝트는 개인 단위로 수행한다.
- 실제 버스 도착 API 연동은 일정과 API 접근성에 따라 조정한다.
- API 연동이 어려울 경우 mock 데이터를 사용하되, 문서에 그 이유와 한계를 기록한다.
- 최종 산출물에는 구현 화면 캡처와 프로세스 기반 논술 PDF가 포함되어야 한다.
- GitHub issue, commit, 회의록을 통해 프로젝트가 주기적으로 진행되었음을 증빙한다.

## 위험 요소

| 위험 | 영향 | 대응 방안 |
| --- | --- | --- |
| 외부 API 연동 지연 | 버스 또는 날씨 기능 구현 범위 축소 가능 | mock 데이터 기반 프로토타입을 우선 구현한다. |
| 기능 범위 과다 | 설계, 테스트, 보고서 작성 시간이 부족할 수 있음 | Must 요구사항을 우선 구현한다. |
| 문서화 누락 | 프로세스 적용 증빙이 약해질 수 있음 | 작업일마다 회의록과 commit을 남긴다. |
| UI 복잡도 증가 | 한 화면에서 정보가 산만해질 수 있음 | 대시보드 영역을 명확히 나누고 핵심 정보만 우선 표시한다. |
