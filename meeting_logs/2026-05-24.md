## 요구사항 분석 논의

외출 준비 통합 도우미 앱은 사용자가 바쁜 아침 시간에 여러 앱을 오가야 하는 불편을 줄이기 위해 기획되었다. 오늘 논의에서는 단순히 기능 목록을 나열하는 대신, 실제 사용 시나리오를 기준으로 요구사항을 정리하기로 했다.

핵심 사용 흐름은 다음과 같이 정리했다.

1. 사용자가 앱을 실행한다.
2. 대시보드에서 오늘의 준비 상태를 확인한다.
3. 준비물 투두를 확인하고 완료 처리한다.
4. 현재 날씨와 기온을 확인한다.
5. 기온별 옷차림 추천을 확인한다.
6. 등록된 버스 도착까지 남은 시간을 확인한다.
7. 설정된 외출 시각까지 남은 시간을 확인한다.

이 흐름을 기준으로 대시보드, 준비물, 날씨, 옷차림, 버스, 외출 시각 기능을 주요 요구사항으로 정리했다.

## 우선순위 결정

기능 우선순위는 Must, Should, Could, Won't 기준으로 나누기로 했다.

- Must: 앱의 목적 달성을 위해 반드시 필요한 기능
- Should: 핵심 사용 경험을 보강하지만 일정에 따라 조정 가능한 기능
- Could: 시간이 남을 경우 구현할 수 있는 기능
- Won't: 이번 프로젝트에서는 제외할 기능

Must 요구사항은 대시보드 화면, 준비물 표시 및 체크, 날씨와 기온 표시, 기온별 옷차림 추천, 외출 시각 및 남은 시간 표시로 정했다. 버스 도착 정보는 핵심 아이디어에 포함되지만 실제 API 연동 가능성이 변수이므로 Should로 두고, 필요하면 mock 데이터로 구현하기로 했다.

## 기능 범위 조정

오늘 논의에서 범위를 다음과 같이 정리했다.

### 포함할 기능

- 외출 준비 대시보드
- 준비물 투두 리스트
- 현재 날씨와 기온 표시
- 기온별 옷차림 추천
- 버스 도착 정보 표시
- 외출 시각 및 남은 시간 표시

### 제외하거나 후순위로 둘 기능

- 로그인 기능
- 사용자 간 데이터 공유
- 복잡한 알림 기능
- 장기 날씨 예보 분석
- 실제 교통 API 완전 연동

실제 API 연동은 프로젝트 일정과 API 접근성에 영향을 받기 때문에, 프로토타입에서는 mock 데이터를 우선 고려하기로 했다. 이 결정은 구현 가능성을 높이고, 요구사항 분석부터 테스트까지의 프로세스를 안정적으로 진행하기 위한 범위 관리 결정이다.

## GitHub Issue 계획

GitHub issue는 요구사항, 설계, 구현, 테스트, 회고가 서로 연결되도록 작업 단위별로 등록하기로 했다. 오늘은 실제 등록에 사용할 issue 초안을 `docs/github_issues.md`에 정리했다.

등록할 issue 후보는 다음과 같다.

- 요구사항 분석 문서 작성
- 화면 및 데이터 구조 설계
- 대시보드 UI 구현
- 준비물 투두 리스트 기능 구현
- 날씨 및 기온별 옷차림 추천 기능 구현
- 버스 도착 정보 표시 기능 구현
- 외출 시각 카운트다운 기능 구현
- 테스트 케이스 및 테스트 보고서 작성
- 프로세스 적용 lessons learned 정리
- 최종 PDF 보고서 작성

각 issue에는 작업 목적, 완료 조건, 관련 label과 milestone을 함께 기록하여 이후 commit과 연결할 수 있도록 한다.
