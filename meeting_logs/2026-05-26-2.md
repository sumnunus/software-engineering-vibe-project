## 진행 배경

2026-05-26 새벽에는 설계 문서 작성을 진행했고, 이후 같은 날 아침에는 Issue #3에 해당하는 대시보드 UI 구현을 시작했다. 두 작업은 같은 날짜에 진행되었지만 성격이 다르므로, 설계 회의록과 구현 회의록을 분리하여 기록하기로 했다.

## 구현 전 결정 사항

대시보드 UI 구현 전 다음 사항을 확인했다.

- 브랜치 전략: main 브랜치가 아닌 `feature/dashboard-ui` 브랜치에서 작업한다.
- 기술 스택: React + Vite 기반 웹앱으로 구현한다.
- 색상 팔레트: Toss Blue Clean 팔레트를 적용한다.
- 로고 방향: `Ready`는 검정, `Go`는 파란색으로 강조하는 워드마크를 사용한다.
- 구현 범위: 실제 기능 완성보다 대시보드 화면 구조와 mock 데이터 기반 표시를 우선한다.
- 중간 검토: UI는 수정 가능성이 높으므로 커밋 전 프로젝트 수행자가 화면을 확인하고 수정 의견을 반영한다.

## 구현 범위

오늘 구현한 범위는 다음과 같다.

- React + Vite 기반 웹앱 구조 생성
- `ReadyGo` 워드마크 로고 적용
- Toss Blue Clean 색상 팔레트 적용
- 남은 시간, 준비물 투두, 오늘 날씨, 관심 정류장 영역 배치
- mock 데이터 기반 대시보드 표시
- 날씨/버스 데이터 교체를 고려한 service 폴더 생성
- `outfitRules`, `timeUtils` 등 계산 로직 분리
- `node_modules/`와 `dist/`가 커밋되지 않도록 `.gitignore` 추가

## 중간 검토 및 수정 사항

브라우저에서 1차 UI를 확인한 결과, 초기에는 빈 화면이 표시되었다. 원인을 확인한 결과 `App.jsx`에서 React import가 누락되어 런타임 오류가 발생한 것이었다. `React` import를 추가하여 빈 화면 문제를 해결했다.

이후 프로젝트 수행자의 화면 검토를 통해 다음 수정 의견을 반영했다.

- 남은 시간 영역에서 외출 시각의 역할이 명확하지 않아 준비 게이지를 추가했다.
- 외출 시각에 가까워질수록 준비 게이지가 차도록 설계했다.
- 상단 소개 문구를 제거하여 화면을 더 간결하게 정리했다.
- 관심 정류장 정보를 `관심 정류장` 제목과 `변경` 버튼 사이에 배치했다.
- 버스 정보 영역은 최대 4개 슬롯을 표시하도록 변경했다.
- 버스 정보가 4개 미만일 경우 빈 슬롯을 표시하도록 했다.
- 준비 게이지 강조 색상은 여러 후보를 검토한 뒤 최종적으로 `#FFE500`을 적용했다.

## 오늘의 교훈

바이브코딩으로 빠르게 UI를 만들 수 있었지만, 프로젝트 수행자가 실제 화면을 확인해 보니 정보 위계와 색감은 코드 작성 전 예상과 다르게 느껴지는 부분이 있었다. 따라서 구현 단계에서도 중간 검토를 통해 수정 의견을 반영하는 과정이 중요하다는 점을 확인했다. 또한 설계 문서에서 정의한 모듈 분리 방향에 맞추어 UI, mock 데이터, 계산 로직을 나누어 두면 이후 기능 구현으로 자연스럽게 확장할 수 있다는 점을 확인했다.
