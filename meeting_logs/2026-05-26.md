## 설계 전 확인한 의도

오늘 설계 전 다음 사항을 확인했다.

- 앱 형태: 웹앱
- 앱 이름: ReadyGo
- 대시보드 정보 우선순위: 남은 시간, 투두, 날씨, 버스 순서
- 디자인 분위기: Toss처럼 깔끔하고 트렌디한 생활 도구형 UI
- 날씨 데이터: 가능하면 실제 API 연동, 복잡해질 경우 mock 데이터 허용
- 버스 데이터: 실제 API를 희망하지만 복잡도가 높으면 mock 데이터 허용
- 투두 기능: 추가, 삭제, 내용 수정, 체크, 체크해제 지원
- 외출 시간 기능: 사용자가 외출 시간을 수정할 수 있고, 남은 시간이 그에 맞게 업데이트되어야 함
- 버스 기능: 관심 정류장을 수정하고, 관심 등록한 정류장의 버스 정보를 보여주는 방식
- 설계 깊이: 과제 제출용으로 충분히 자세하게 작성
- 설계 원칙: 추상화, 캡슐화, 모듈화, 결합도와 응집도, SOLID 객체지향 설계 원리 반영

## 주요 설계 논의

### 화면 구조

ReadyGo는 단일 대시보드 화면으로 설계했다. 사용자가 아침에 가장 먼저 확인해야 하는 정보가 외출까지 남은 시간이므로 화면 최상단에 배치하고, 그 아래에 준비물 투두를 둔다. 날씨와 버스 정보는 보조 판단 정보로 하단에 배치한다.

대시보드 영역은 다음 네 가지로 나누었다.

- 남은 시간 영역
- 준비물 투두 영역
- 날씨 및 옷차림 영역
- 버스 정보 영역

### 데이터 연동 전략

날씨는 실제 API 연동을 우선 검토하기로 했다. 후보로는 Open-Meteo를 선택했다. Open-Meteo는 API key 없이 사용할 수 있고 현재 기온, 체감 온도, 강수량, 날씨 코드 등을 제공하므로 초기 웹앱에 적합하다고 판단했다.

버스 도착 정보는 실제 공공 API 연동을 희망하지만, 인증키와 정류장 ID, 지역별 API 구조 등 고려할 요소가 많아 초기 구현에서는 mock 데이터를 우선 사용하기로 했다. 대신 `busService` 구조를 설계하여 추후 실제 API로 교체할 수 있게 한다.

### 소프트웨어공학 설계 원칙

이번 설계에서는 수업에서 배운 소프트웨어공학 원칙을 명시적으로 반영하기로 했다.

- 추상화: 실제 API와 mock 데이터를 동일한 데이터 형태로 다루도록 설계
- 캡슐화: API 호출, localStorage 접근, 시간 계산, 옷차림 추천 규칙을 별도 모듈에 숨김
- 모듈화: 남은 시간, 투두, 날씨, 버스 영역을 독립 컴포넌트로 분리
- 낮은 결합도: UI가 특정 API 응답 형식에 직접 의존하지 않도록 service 계층에서 변환
- 높은 응집도: 각 모듈이 하나의 책임만 갖도록 구성
- SOLID: 단일 책임 원칙, 개방-폐쇄 원칙, 리스코프 치환 원칙, 인터페이스 분리 원칙, 의존성 역전 원칙을 설계 문서에 반영

## 결정 사항

- 앱 이름은 `ReadyGo`로 한다.
- 웹앱 형태로 구현한다.
- 화면은 단일 대시보드 구조로 시작한다.
- 정보 우선순위는 남은 시간, 투두, 날씨, 버스 순서로 한다.
- 날씨 API는 Open-Meteo를 후보로 실제 연동을 우선 검토한다.
- 버스 정보는 초기 구현에서 mock 데이터를 우선 사용하되, 실제 API 교체 가능성을 열어둔다.
- 사용자 입력 데이터인 투두, 외출 시각, 관심 정류장은 localStorage 저장 대상으로 설계한다.
- 외부 조회 데이터인 날씨와 버스 도착 정보는 화면 상태로 관리한다.
- API와 mock 데이터는 service 계층으로 추상화한다.

## 오늘의 교훈

설계 단계에서 UI와 API를 바로 연결하려고 하면 구현은 빨라 보이지만, 이후 API 변경이나 mock 데이터 전환에 취약해질 수 있다. 반대로 service 계층을 두고 데이터 구조를 먼저 정리하면 실제 API와 mock 데이터를 교체하기 쉬워진다. 또한 추상화, 캡슐화, 모듈화 같은 원칙을 문서에 명시하니 단순한 화면 설계가 아니라 유지보수성과 테스트 가능성을 고려한 설계로 확장되었다.
