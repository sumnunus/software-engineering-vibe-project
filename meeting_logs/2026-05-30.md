## 구현 전 확인한 의도

기능 구현 전 프로젝트 수행자의 의도를 다음과 같이 확인했다.

- 실제 서울 버스 API 연동을 우선 시도하고, 기술적 제약이 있으면 mock 데이터로 전환한다.
- 관심 정류장은 정류장 이름 검색으로 변경한다. 실제 검색이 어려우면 미리 준비한 목록을 제공한다.
- 선택한 관심 정류장은 localStorage에 저장하여 새로고침 후에도 유지한다.
- 버스 번호, 도착까지 남은 시간, 혼잡도를 표시한다.
- 도착 정보는 30초마다 자동 갱신하고, 사용자가 직접 누를 수 있는 새로고침 버튼도 제공한다.
- 표시할 버스가 4개 미만이면 남은 칸에 `대기 중`을 표시한다.

## 공식 API 가이드 검토

공공데이터포털에서 발급받은 `서울특별시 정류소정보조회 서비스` 인증키와 활용가이드를 기준으로 API 구조를 확인했다.

- 정류장 이름 검색: `getStationByName`
- 정류장별 실시간 도착 정보: `getStationByUid`
- 주요 응답 필드: 정류장명, 정류소 번호, 노선명, 방향, 도착 메시지, 혼잡도
- 기술적 고려사항: API가 HTTP 기반이므로 HTTPS 배포 환경의 브라우저에서 직접 호출하면 mixed content 또는 CORS 문제가 발생할 수 있다.

## 소프트웨어공학 원칙 적용 방향

버스 API의 기술적 제약이 UI에 영향을 주지 않도록 책임을 분리한다.

- `busService`: 실제 API 요청, 응답 정규화, mock fallback 정책을 캡슐화한다.
- `mockBusService`: API를 사용할 수 없을 때 예시 정류장과 도착 정보를 제공한다.
- `storageService`: 사용자가 선택한 관심 정류장을 localStorage에 저장하고 복원한다.
- `vite.config.js`: 개발 환경에서 인증키를 노출하지 않고 API를 호출할 수 있도록 proxy를 설정한다.
- `BusPanel`: 검색, 정류장 선택, 새로고침과 화면 표시를 담당한다.

이 구조를 통해 실제 API 사용이 어려운 환경에서도 화면 기능을 유지하고, 추후 별도의 backend proxy를 도입할 때 UI 변경을 최소화할 수 있다.

## 중간 검토 및 수정 사항

브라우저에서 검색 UI를 검토하며 다음 의견을 반영했다.

- `변경` 버튼은 관심 정류장 검색 영역을 여닫고, 원형 화살표 버튼은 도착 정보를 즉시 갱신하도록 역할을 구분했다.
- 정류장 검색 결과에서 개발자용 정보처럼 보이는 정류소 번호는 기본적으로 숨겼다.
- 이름이 같은 정류장이 여러 개 검색되어 구분이 필요한 경우에만 정류소 번호를 보조 정보로 표시한다.
